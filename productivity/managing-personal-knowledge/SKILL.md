# Managing Personal Knowledge

## Scope
Designing and running a personal knowledge management (PKM) system across capture,
organization, linking, and retrieval — the "second brain." Covers the system and
its habits, not the moment-to-moment craft of writing a single note.

## Purpose
Make what you learn findable and reusable later. Turn scattered notes, bookmarks,
and highlights into a trusted system you actually pull from when doing real work.

## When to use this skill
- Notes, highlights, and bookmarks accumulate but can't be found again.
- Setting up or overhauling a note vault, personal wiki, or knowledge base.
- Choosing an organization scheme (PARA, topic folders, tags, links).
- Past research keeps getting lost or re-done from scratch.

## When NOT to use this skill
- How to write a good note in the moment → [taking-notes](../taking-notes/SKILL.md).
- Combining sources into a written argument → [synthesizing-research](../../research/synthesizing-research/SKILL.md).
- Org-wide data governance, retention, or shared systems → out of scope (a team/IT concern).

## Inputs
- Existing notes, highlights, bookmarks, files — however scattered.
- The tools in play (app, files, wiki) and how you actually work.
- The kinds of questions you expect to ask the system later.

## Outputs
- A capture-to-retrieval workflow with a clear inbox and a home for finished notes.
- An organization scheme (e.g. PARA) plus linking/tagging conventions.
- A review cadence that keeps the system trustworthy over time.

## Workflow
```
Progress:
- [ ] 1. Define what the system must answer and pick your tools
- [ ] 2. Set up one frictionless capture inbox
- [ ] 3. Choose an organization scheme (e.g. PARA) and a home for notes
- [ ] 4. Establish linking and light tagging conventions
- [ ] 5. Make retrieval work: search, links, and index/MOC notes
- [ ] 6. Set a review cadence to process the inbox and prune
```

**Step 1 — Define and choose.** Name the questions the system should answer later
and pick tools that fit how you work. Optimize for retrieval, not collection.

**Step 2 — Capture.** Set up one low-friction inbox for everything. Capture fast and
sort later; a note you don't record is lost, but an unsorted note is recoverable.

**Step 3 — Organize.** Adopt a scheme — PARA (Projects, Areas, Resources, Archives)
is a strong default: organize by actionability, not by topic taxonomy. Give
finished notes a stable home out of the inbox.

**Step 4 — Link.** Connect related notes with links; use a few broad tags for
cross-cutting themes. Links beat deep folder trees for surfacing connections.

**Step 5 — Retrieve.** Make things findable: good titles, reliable search, and
index/map-of-content (MOC) notes that gather links to a topic's key notes.

**Step 6 — Review.** Schedule a recurring pass to empty the inbox, file or link new
notes, and archive the stale. Without this, the system decays into a graveyard.

## Principles
1. **Retrieval is the point** — capturing you never revisit is hoarding.
2. **Organize by actionability** (PARA), not an ever-growing topic tree.
3. **One inbox, low friction** — capture now, organize on review.
4. **Links over deep folders** — connection surfaces reuse.
5. **A few tags, used consistently** — tags are for cross-cutting themes, not everything.
6. **A trusted system** — kept current so you rely on it instead of your memory.

## Decision framework
- **Where does a note go?** Ask "which project/area does this serve?" before topic.
- **Folder or tag or link?** Home = one folder; themes = tags; relationships = links.
- **Tag or not?** Only if you'd browse by it later; otherwise skip it.
- **Keep or archive?** If it hasn't been useful and likely won't be, archive it.
- **System feels heavy?** Simplify the scheme; friction kills the habit.

## Common mistakes
- **Knowledge hoarding** — collecting endlessly with no retrieval or review.
- **Over-tagging** — dozens of one-off tags that fragment rather than organize.
- **Elaborate folder trees** — deep topic hierarchies too ambiguous to file within.
- **No inbox / no processing** — captures scattered everywhere, never filed.
- **Tool-hopping** — migrating apps instead of building the habit.
- **Orphan notes** — nothing links to them, so nothing surfaces them.

## Validation checklist
- [ ] Every capture lands in a single known inbox.
- [ ] Finished notes have a stable home under a clear scheme (e.g. PARA).
- [ ] Related notes are linked; tags are few and consistent.
- [ ] A recent test question was answered from the system in seconds.
- [ ] A review cadence exists and the inbox is regularly emptied.
- [ ] Stale material is archived, not left cluttering active space.

## Edge cases
- **Legacy pile of notes:** don't reorganize all at once — process on demand plus a small daily quota.
- **Multiple tools/devices:** pick one source of truth; others feed the inbox.
- **Sensitive information:** keep secrets out of personal notes; reference, don't copy.
- **Shared/team knowledge:** a personal system is not a team wiki — link out instead of duplicating.

## Examples
**Input:** "My notes are spread across three apps and I can never find anything."
**Output:** Chose one app as source of truth; the others now feed its inbox. Adopted
PARA: active work under Projects, ongoing responsibilities under Areas, reference
under Resources, done under Archives. Added MOC notes per major area linking key
notes; trimmed 40 ad-hoc tags to 8 themes. Weekly 20-minute review empties the
inbox. Test: "that pricing article" found by search in five seconds.

## Automation opportunities
- Use a web clipper / share-sheet that drops captures straight into the inbox.
- Auto-import highlights from your reader/e-book app.
- Use backlinks and saved searches to auto-surface related notes and MOCs.
